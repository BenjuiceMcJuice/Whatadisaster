## Pixel-art graphics overhaul — fire maps + UI icons (branch `claude/app-graphics-animations-x72ur9`)

Stakeholder ask: the graphics/animations looked "a bit lame" — could they be better, or is that a limit of the single-HTML-file format? Answer: not a format limit. Reworked the app's hero visuals in a stylised-but-crisp pixel-art direction (dramatic, per stakeholder steer), no new dependencies, still one HTML file.

**Fire-map interstitial rebuilt as a `<canvas>` engine (`FMAP`)**
- Replaced the two growing-ellipse SVGs (`svgAshdown` / `svgCabot` + `fire*`/`emb*`/`cfire*`/`cemb*` shapes) with a single `<canvas id="fireCanvas">` (256×141 internal, upscaled with `image-rendering:pixelated` → crisp chunky pixels at any size).
- Flames driven by the classic propagating-fire cellular automaton ("doom fire"): per-frame seed row from each `MAPS`/`MAPS2` fire zone, upward propagation with random decay + wind drift, mapped through a 37-step palette. Particles layer on top (rising embers + smoke for wildfire; heat shimmer for heatwave).
- Two faithful pixel base-maps drawn to an offscreen buffer (`drawAshdown` / `drawHeat`) matching the old SVG layouts (Brynmoor Hill, B4107, M5, Ashfield, Meridian, substation, Northgate; city centre, high-rises, school, care home, sun).
- **Ashdown** = tall orange flames billowing off the hill (dramatic). **Solstice** = retuned to a low, dimmer, translucent amber glow hugging rooftops/estates + shimmer — reads as *extreme heat radiating*, not a city on fire (first tuning looked like a wildfire; fixed via `seedMax`/`seedProb` + higher decay + full-height fade).
- `prefers-reduced-motion`: renders a single developed static frame, no RAF loop. `FMAP.stop()` cancels the loop on Continue / re-entry (no leaked animation frames).

**Pixel-art icon system (`PX`)** — replaces prominent emoji in the UI chrome
- Crisp inline-SVG sprites (one `<rect>` per pixel, `shape-rendering:crispEdges`), defined as char-grid + colour map, or generated (sun). `PX.icon(name,{size})` returns SVG; `PX.hydrate()` fills any `[data-pxicon]` element on load.
- Swapped: splash hero 🚨 → pixel emergency beacon; scenario buttons 🔥/☀️ → pixel flame/sun; briefing button + header + disclaimer link 📋 → pixel document (folded corner + orange text lines — first pass read as a battery, redesigned); dev-warning ⚠️ → pixel warning triangle; fire-map fact label 🔥/☀️ → flame/sun.
- Left as-is (scoped to chrome, not content): inline emoji inside question/feedback prose, the small agency chips (🚒🚔🚑🏥🏛️⚡), press ⚡, breakdown 📊, accordion 📌🎯 — these are small inline labels, not "graphics".

**Verification** (headless Chromium, `/opt/pw-browsers`): splash, both fire maps, briefing, and reduced-motion all render with **0 JS errors** (only sandbox-blocked fonts/backend). No dangling references to the removed SVG ids. Not yet merged/deployed — lives on the branch.

### Pivot: propagating-fire → sprite-based tactical operational map

Stakeholder refined the brief (reference: live map screenshot) toward a **professional incident map**, not retro game art: flat shapes, limited palette, modular + reproducible, fire that grows by *adding* clusters rather than a simulation. Replaced the `FMAP` doom-fire engine with a sprite/tactical model.

- **Static base map** drawn once to an offscreen buffer (terrain, faint GIS grid, pixel trees, grey roads, flat buildings with 2–4 windows + doorway, cyan-highlighted assets, yellow infra, hospital cross, sun). Only the incident layer animates → cheap and calm ("control-room display").
- **Wildfire (Ashdown):** flat flame sprites + translucent chunky spread-zone discs + ember spot-fires, all seeded from `MAPS[phase].fires/embers`. Fire grows each stage purely by adding clusters/expanding discs; map bytes are identical across stages. Flames flicker on a 2-frame cycle (only motion); embers pulse.
- **Heatwave (Solstice):** no literal flames — heat rendered as **upward wavy lines ramping amber → deep red** as the alert intensifies (colour keyed to stage index), rising from each at-risk building/zone in `MAPS2`, over faint amber heat discs.
- **Labels drawn last, every frame, with a 1px dark backing** so they stay readable over the fire (fixes the earlier overrun); `NORTHGATE` edge-clip fixed.
- `FMAP.render(scenario, m, idx)` / `FMAP.stop()` interface unchanged; `showFireMap()` now passes the stage index. Respects `prefers-reduced-motion` (single static frame, no RAF).

**Verified** (headless Chromium, deviceScaleFactor 3): both scenarios × all 3 stages render from the live app with **0 JS errors**; fire visibly grows per stage, heat ramps amber→red, labels legible.

### Heatwave rework: wavy lines → site-severity impact board

Stakeholder feedback: the wavy red heat lines didn't work, and — more fundamentally — the heatwave infographic was weak because *heat doesn't spread like fire*. Reframed what escalates: not spatial spread, but **cumulative impact on vulnerable sites**, peaking at the tropical night (when temperature actually *falls* but risk peaks — the key exercise lesson). A literal thermometer would mislead (stage-3 night reads "cooler"); so the gauge tracks heat-health *risk*, not temperature.

New heatwave rendering in `FMAP` (wildfire path untouched):
- Each vulnerable site (care home, Meridian/Northgate high-rises, school, hospital ED, Ashfield estate) gets a **status-coloured building outline** (green monitored → amber at risk → red critical) plus a **thermometer-style gauge** that rises and reddens. Building outline = at-a-glance status; gauge = the reading.
- **Escalating alert badge** (`LEVEL 4` + state) with a **time · temp** readout: `15:00·40C` → `21:00·32C` → `02:00·28C`, so falling-temp / rising-risk is explicit.
- Severity keyed to stage index (`FMAP.render(... , idx)`): stage 1 mostly amber, stage 3 a board of red. **School de-escalates to green at night** (children home) — realistic, and keeps the board readable rather than uniformly red.
- Critical (red) sites pulse; everything else static (same motion rule as wildfire). Reduced-motion shows a static frame with all rings lit.
- Simplified heat map labels (CARE HOME / SCHOOL / HOSPITAL) and dropped the stray 39C sun label (temp now lives in the badge).

**Verified** (headless, dsf 3): both scenarios × 3 stages render from the live app, 0 JS errors; wildfire grows, heatwave board escalates amber→red with the night twist.

### High-res rebuild — fixes low-res look, unreadable labels, weak beacon, cryptic thermometers

Live-preview feedback (with a ChatGPT reference the stakeholder preferred): the maps looked low-res, map text was barely readable, the splash "light" was weak, and the heatwave thermometers were unclear. Root cause of the first two: the engine drew everything — **including text** — onto a 256px canvas then upscaled ~4x with `image-rendering:pixelated`, which destroys text and looks low-res.

Re-architected `FMAP` to render at **1024×576 with smooth downscale** (removed `image-rendering:pixelated`; canvas backing store bumped to 1024×576). Pixel feel now comes from chunky blocks + limited palette, not from blowing up a thumbnail.
- **Real-font labels** (Barlow Condensed) drawn at full resolution → crisp and readable.
- **Wildfire**: detailed block flames + rising smoke, conifer trees, buildings with a curved estate road network, soft radial burn-glow that grows with the fire; flame/smoke/ember positions data-driven from `MAPS[phase]` so the fire grows across the 3 stages.
- **Heatwave**: replaced the unreadable thermometer gauges with **text status tags** (`MONITORED` / `AT RISK` / `CRITICAL`) + status-coloured building outlines; vulnerable building bodies now drawn in the base (care home, high-rises, hospital w/ cross, school, estate), so critical sites stay visible while their outline+tag pulse (glow), rather than vanishing on the blink. Escalating alert badge + time·temp retained. School de-escalates to green at night.
- **Splash beacon** redesigned at 24×24 with a glossy red dome, amber lens band, dark base and light flashes (was a flat 14px blob).

**Verified** (headless, dsf 2–3): both scenarios × 3 stages, splash beacon, and reduced-motion (static frame, critical sites visible) all render with **0 JS errors**.
