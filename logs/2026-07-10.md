**Files changed:** `index.html`, new `firestore.rules`

**What was added to `index.html`:**
- Firebase SDK via CDN as an ES module (`firebase-app.js` / `firebase-firestore.js` v10.13.0) — no build step, consistent with the single-file architecture. `firebaseConfig` excludes `measurementId`/Analytics.
- `window.submitFeedback(payload)` and `window.logEvent(type, extra)` exposed from the module script for the existing ES5 code to call.
- Persistent **Feedback button** (`.fb-toggle`, fixed bottom-right, all screens) opening a bottom-sheet modal (`#feedbackModal`, reuses the `.mo`/`.md` modal pattern already used for decision feedback) — category select (content/UX/feature/bug) + free-text, writes to the `feedback` Firestore collection with `source: 'community'`, `status: 'open'`, auto-attached `role`/`scenario` when in a game.
- Usage event logging (`events` collection) wired into the existing flow, no new UI:
  - `role_selected` — `selectRole()`
  - `scenario_started` — `startGame()`
  - `question_answered` — `pick()`, includes `qIdx`
  - `scenario_completed` — `endGame()`, includes `score`
  - `feedback_submitted` — on successful feedback write
  - `logEventSafe()` wrapper fails silently (try/catch, drops falsy fields) so a blocked/slow Firebase load never breaks gameplay.

**`firestore.rules`** (new file, not yet CLI-deployed — no Firebase CLI configured, so this is the source of truth to manually paste into Console → Firestore → Rules → Publish):
- `feedback`: create-only, validates required fields, locks `source` to `'community'`, `category` to a fixed enum, description length capped at 2000 chars, `createdAt` must equal `request.time` (blocks backdating). No client read/update/delete.
- `events`: create-only, `type` restricted to the five known event names, optional `qIdx`/`score` type/range-checked. No client read/update/delete.
- Rate limiting (Firestore rules can't do this alone) is a known follow-up — App Check or a Cloud Function would be the next step if abuse becomes a problem.
