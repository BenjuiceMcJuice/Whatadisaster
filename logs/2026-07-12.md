## Branching review — does a bad choice actually change anything?

Trigger: Ben asked whether the game's branching (`branch_sit`/`branch_flag`) actually makes a difference when you pick a bad option.

**Method:** traced `pick()` → `G.flags` → `applyBranching()` → `getBranchPressure()` → `endGame()` end-to-end for all 4 defined flags (`noTCG`, `towerDelayed`, `hospitalLate`, `registerWithheld`) across every role in both `QB` and `QB2`, rather than just reading the data model.

**Finding:** score and the 4 consequence meters (`G.cons`) always respond correctly to every choice, in every role/scenario, and correctly drive the final outcome band. But the `branch_sit` situation-text rewrite — the mechanism meant to show a grimmer scene after an early bad call — is **dead code in all 4 cases**: 3 are self-referential (the flag can't be set until *after* the very question that would display the rewrite), and the 4th (`noTCG`) is orphaned across roles (set only by `tactical`, read only by `strategic` — the two never coexist in one playthrough). Two flags (`noTCG`, `hospitalLate`) still produce a working fallback (a warning chip on later questions, and for `noTCG` a debrief line); `registerWithheld`'s chip works but is imprecisely triggered by any "Vulnerable Persons"-titled question; `towerDelayed` has zero visible effect anywhere. The whole system only exists in `QB` (Ashdown) — `QB2` (Solstice) has no branch points at all.

Confirmed via a full `G.flags` usage audit (6 read/write sites total, `index.html`) that none of this affects available options, question sequence, or scoring — the impact is purely on narrative/feedback text, not gameplay content.

**Deliverable:** `info/branching-review-2026-07-12.md` — no code changed, review only.

## Narrative-tone spec — replacement design

Following on from the review above: Ben wants a bad/good run to read differently without over-complicating things or requiring per-question authoring, and it should work across replays generically ("as much as is reasonable").

**Approach chosen over scaling up the old flag system:** drive situation-text tone off the existing `G.cons` meters (already continuous, already identical across all roles/scenarios) rather than hand-authored per-question branches. Spec covers three parts:

- **Cleanup** of the 4 dead `branch_sit`/`branch_flag` pairs and `applyBranching()`, keeping the 3 chip/debrief fallbacks that already work, and a one-line fix so `towerDelayed`'s chip condition actually fires.
- **New mechanism:** `getNarrativeTone()` reuses the existing severity bands from `updateConsBars()` (Controlled/Manageable/Significant/Serious/Critical) to append a short tone clause to `sit` text — silent below Significant, generic multi-meter clause or a named-meter clause (whichever meter is the clear outlier) above it. ~18 short sentences total is the entire content cost; no per-question or per-role authoring.
- **Debrief generalization:** extend the existing `noTCG`-only debrief callout pattern to name whichever meter ended worst, for any role/scenario.

**Deliverable:** `info/narrative-tone-spec-2026-07-12.md` — full data structures, hook points (`loadQ()`, `endGame()`), content-bank sizing, and a test plan. Not yet built. Added to DEVLOG backlog (**High**) with an explicit note that once it ships, a standalone "how it works" reference doc must be written (the spec is the build plan, not the lasting reference).
